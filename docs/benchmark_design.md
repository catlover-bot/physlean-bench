# Benchmark Design

## Scope and Positioning

`physlean-bench` is specialized for Physlib-derived Lean 4 benchmark construction from `leanprover-community/physlib`.

- Phase priority: Physlib side first.
- QuantumInfo: preserved as future extension, but excluded from release-candidate path in this phase.

## Authoritative vs Fallback Pipelines

### Authoritative pipeline (`leandojo_v2`)

This is the benchmark-release path.

Stages:
1. clone/pin source,
2. cache fetch/build,
3. trace with LeanDojo-v2,
4. validate trace provenance/structure,
5. traced-only inventory extraction/filtering,
6. traced-only completion extraction,
7. split generation,
8. audit and release packaging.

### Fallback pipeline (`source_scan`)

This is a real, non-mock source parser path used for development when full tracing is blocked.

- Explicitly marked as `source_scan_fallback` in metadata.
- Not authoritative for release-candidate claims.

## Preflight and Operational Robustness

Preflight checks now run before tracing and record machine-readable + markdown artifacts:
- repository presence and git status,
- Python/tooling availability,
- `lake`/Lean resolution,
- `lean_dojo` package availability,
- free disk threshold,
- output writability.

Tracing writes stage markers/checkpoints under `trace/stages/` and supports resume when outputs already exist.

## Trace Validation and Provenance

`validate-trace` verifies:
- source URL/commit consistency,
- backend consistency (`leandojo_v2` for traced release path),
- non-empty/sane record counts,
- include/exclude path constraints,
- metadata completeness.

Outputs:
- `trace_validation.json`
- `trace_validation.md`

## Inventory Policy

Inventory entries include:
- declaration name, namespace, module/file path,
- statement/proof text where recoverable,
- premise metadata (accessible/used if recoverable),
- provenance fields (`trace_backend`, `source_url`, `source_commit`),
- filtering outcome and exclusion reason,
- quality flags/metrics.

Filtering excludes (default):
- missing statement/proof,
- `sorry`/`admit`/placeholder patterns,
- internal/autogenerated declarations,
- non-theorem-like declarations,
- duplicate statements.

Optional stricter controls exist for likely trivial or very short proofs.

## Quality-Control Heuristics

Heuristics are practical and non-perfect, with explicit metadata retention:
- very short proofs,
- likely `rfl`-only / `simp`-only trivial proofs,
- duplicate statement candidates,
- missing premise metadata,
- namespace/file concentration summaries.

These heuristics support manual review and do not claim complete detection.

## Completion Dataset Design (Traced-First)

Each completion example carries:
- theorem name and ids,
- namespace/module/file metadata,
- import/header context,
- theorem prompt with `by sorry`,
- gold proof text when recoverable,
- accessible/used premises,
- provenance + quality/audit metadata.

## Split Design

Implemented split strategies:
- `random`
- `file`
- `namespace`
- `novel_local_premise`

Split profiles:
- `small`
- `dev`
- `release_candidate`

Novel-local-premise split prefers traced local-premise information when present and emits caveats when premise coverage is missing.

## Audit and Human Review

`audit-completion` provides:
- sampled examples by namespace/file/difficulty/suspicion filters,
- excluded-by-reason summaries,
- export to JSON + Markdown,
- comparison report against source-scan datasets (when both are available).

Key outputs:
- `audit_sample.json`
- `audit_sample.md`
- `comparison_traced_vs_source_scan.md`

## Release-Candidate Criteria

A release candidate should include:
- traced backend provenance (`leandojo_v2`),
- successful trace validation,
- traced-only inventory/completion,
- split assignments + split summary,
- audit outputs,
- hashes/manifests/config snapshots under release packaging layout.

If these conditions are not met, outputs should be labeled as dev/fallback artifacts, not benchmark-release artifacts.
